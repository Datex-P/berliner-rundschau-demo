## Architectural decisions

### Routing

File-based routing via the Next.js App Router. All pages under `src/app/`. Dynamic segments in square brackets (`[slug]`). Route parameters are validated with Zod (`src/lib/schemas.ts`) and never blindly cast as `string`.

### Data layer

Three-layer architecture:
1. **`mock.ts`** — Transport layer: returns inline data (no external API calls currently)
2. **`data.ts`** — Business layer: uses Next.js `"use cache"` + `cacheLife` + `cacheTag` for selective cache invalidation. All functions are `server-only`.
3. **Pages** — call only functions from `data.ts`, never `mock.ts` directly

Cache tags enable on-demand revalidation via `POST /api/revalidate` (secured with `REVALIDATION_SECRET`).

### CMS migration

To switch from mock to a real CMS, only `mock.ts` needs to be replaced. All other layers (`data.ts`, pages, types) remain unchanged — as long as the return types in `src/types/index.ts` are respected.

### State management

No global state manager. Data is passed as Server Component props. Dark mode state runs through `ThemeProvider` (React Context, Client Component). No server state synchronisation needed.

### Error handling

- Route-level: `error.tsx` per route segment
- Global: `global-error.tsx` as root fallback
- Client-side: `ErrorReporter.tsx` catches `unhandledrejection` events
- All pages with data fetching have an `ErrorState` fallback with a retry button

### SEO

- `metadata` export on every page (OpenGraph, Twitter Card)
- JSON-LD Structured Data via `src/lib/json-ld.ts`
- `robots.ts` generates `robots.txt` dynamically
- Fonts are embedded at build time (no external Google Fonts request at runtime)

### Dark mode

Initialised via inline script in `<head>` (prevents FOUC). Toggle via `ThemeProvider` + `localStorage`. No flash-of-wrong-theme on page load.

### Paywall

Articles have a `paywall` field (`"free" | "paid" | "metered"`). Paywall logic is implemented in the article pages. Actual access control (authentication, token validation) must be added when connecting to a real CMS.

## Environment variables

All variables in `.env.local` (do not commit — already in `.gitignore`):

| Variable | Type | Description | Required |
|----------|------|-------------|----------|
| `REVALIDATION_SECRET` | Secret | Authentication for `POST /api/revalidate` (cache invalidation) | Production |
| `API_BASE_URL` | Config | Base URL of the site (fallback: `localhost:3000` in dev) | Optional |

> `NEXT_PUBLIC_*` variables end up in the client bundle — never prefix secrets with this pattern.

`.env.example` contains all required variables with descriptions.

## Extensibility

### Adding a new page

Create a new file at `src/app/<route>/page.tsx`. Next.js automatically detects it as a route.

### Adding a new component

Place it in `src/components/` or `src/components/ui/` (for primitive elements). Add new data types to `src/types/index.ts`.

### Connecting a CMS

1. Replace `src/lib/mock.ts` with a real API client (keep the same function signatures)
2. Add required environment variables to `.env.local` and the Vercel dashboard
3. Adapt `cacheTag` invalidation in `src/lib/data.ts` to the CMS webhook

### Adding a category to navigation

Edit `src/lib/mock.ts` → `mockNavigation` (after CMS migration: manage in the CMS).

## Known limitations

- **No authentication:** The paywall logic shows UI locks but has no real access control. Authentication must be implemented separately.
- **Mock data:** All content is sample data in `src/lib/mock.ts`. Connect a CMS for production use.
- **CSP with `unsafe-inline`:** Required for Next.js 16 hydration. Migration to nonce-based CSP is recommended after deploy (see `src/lib/security-headers.ts`).
- **No RSS feed:** `feed.xml` is referenced in `metadata.alternates` but not yet implemented.
- **No comment system:** The `Comment` interface and `commentCount` are prepared, but there is no live comment functionality.
